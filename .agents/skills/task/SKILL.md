# /task — выполнить следующий шаг плана

Ведёт **один шаг** активной версии из `docs/tasks/` от анонса до коммита.
Соблюдает парный режим и Conventional Commits из [AGENTS.md](../../../AGENTS.md).

Общайся по-русски (комментарии в коде — на английском).

## Аргументы

- Без аргумента — взять **следующий незавершённый шаг** (первый шаг, где есть `- [ ]`).
- Аргумент = номер шага (`3`) или ID задачи (`v0-4.2`) — работать именно с ним.

## Алгоритм

### 0. Определить активную версию и шаг
1. Прочитать `docs/tasks/README.md`, найти версию со статусом 🚧 → открыть её файл (`docs/tasks/vN-*.md`).
2. Выбрать целевой шаг (по аргументу или первый с незакрытыми `- [ ]`).
3. Прочитать соответствующий раздел в `docs/v0-implementation-plan.md` (или плане активной версии) для контекста и обоснований.

### 1. Анонс (перед работой)
Кратко, по-русски:
- **Что** за шаг и какие подзадачи (ID) в нём.
- **Что конкретно** будем делать.
- **Почему** так (+ альтернатива, если уместно — правило 4 AGENTS.md).
- **Кто что пишет** в этом шаге (см. шаг 2).

### 2. Распределить, кто пишет (правила 2 и 3 AGENTS.md)
Для каждой подзадачи шага определить тип:
- **Рутина** (конфиг, boilerplate, типы, `package.json`, Dockerfile, вёрстка) → **пишет агент**, пользователь ревьюит.
- **Концептуально новое** (SQL-миграция, WebSocket-gateway, `MessagesService`, позже OAuth/Redis Pub/Sub — особенно первый контакт с технологией) → **агент даёт каркас/псевдокод и объясняет концепцию, финальную логику пишет пользователь**, затем агент ревьюит.

### 3. Архитектурные развилки (правило 1)
Если внутри шага всплывает невыбранное архитектурное решение (схема, контракт, выбор библиотеки) —
**остановиться** и показать 2-3 варианта с trade-offs (AskUserQuestion). Не решать молча.

### 4. Выполнение
- Делать строго подзадачи **этого** шага. Не убегать вперёд (правило 5).
- Для setup/конфигурации библиотек и их API — использовать Context7 MCP (resolve-library-id → query-docs) перед написанием.
- Подзадачи, помеченные как «пишет пользователь», — отдать пользователю: дать каркас + объяснение, дождаться его кода, отревьюить. Шаг не считается готовым, пока эти части не написаны и не приняты.
- Где уместно — проверить результат (сборка пакета, запуск, `tsc`).

### 5. Отметить прогресс
- Проставить `- [x]` всем закрытым подзадачам шага в файле версии.
- Если все задачи версии закрыты — обновить её статус в `docs/tasks/README.md` (🚧 → ✅) и пометить следующую версию 🚧.

### 6. Итог и обзор изменений (перед коммитом)
Сначала дать пользователю всё, что нужно для решения о коммите:
- **Итог по-русски:** что сделано в этом шаге, какие подзадачи (ID) закрыты.
- Застейджить только файлы этого шага + изменённый файл задач.
- Показать, что именно уйдёт в коммит, чтобы пользователь сам просмотрел изменения в Source Control:
  - список застейдженных файлов (`git status --short`) и при необходимости дифф (`git diff --staged`),
  - подготовленное сообщение коммита (см. формат ниже).
- Сообщение по **Conventional Commits** (см. AGENTS.md):
  - `type`: `feat` для функциональности, `chore`/`build` для инфры/конфигов, `docs` для документации и т.п.
  - `scope`: `api` · `widget-sdk` · `dashboard` · `shared` · `infra` · `db`.
  - описание — повелительное наклонение, английский, до ~72 символов.
  - в теле перечислить закрытые ID задач, напр. `Closes v0-3.1, v0-3.2, v0-3.3`.

### 7. Коммит (только после разрешения пользователя, без push)
- **Остановиться и спросить разрешение на коммит** (AskUserQuestion: «Закоммитить?»),
  дав пользователю время просмотреть итог и изменения по гиту из шага 6. **Не коммитить, пока он не подтвердил.**
  - Если просит правки (сообщение, состав файлов, сама логика) — внести, снова показать итог + изменения (шаг 6) и опять спросить.
- Закоммитить **только после подтверждения**. **Не делать `git push`** — это отдельное действие.
- Завершить трейлером:
  ```
  Co-Authored-By: Codex Opus 4.8 <noreply@anthropic.com>
  ```
- После коммита коротко сообщить хеш + сообщение и назвать следующий шаг.
- Один вызов `/task` = один шаг = один коммит. Остановиться, не начинать следующий шаг.
